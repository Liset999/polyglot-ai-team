polyglot-ai-team 详细路线图
Claude 风格长文版：架构、控制面、子任务拆分、CLI Worker、飞书移动端、阶段路线图与最终效果
核心判断： 这个项目不是做“模型接入器”，而是做一个 AI 任务执行系统。它通过蓝图生成、角色分配、子任务
拆分、Worker 执行、结果合并、失败重规划、产物回放，把复杂任务变成一条可控、可审计、可复现的流水线。
1. 项目定位
你要做的不是一个普通聊天机器人，也不是一个通用 Agent 框架。你要做的是一个终端优先、支持飞书控制面的多模
型任务执行系统。它接受用户的一句话需求、PRD 或链接，然后自动生成可执行的工作流蓝图，继续把蓝图拆成
Stage，再把 Stage 拆成子任务，交给多个 Worker 并行或串行执行，最后把产物、日志、回放和审计信息全部落盘。
这个项目的本质是：把“人脑里对任务的组织方式”显式化、结构化、可执行化。
2. 为什么要这样做
现实任务不是一条线，而是要拆成设计、实现、测试、修复、整理、汇报等多个阶段。
模型不是一次就能做对，任务执行需要重规划、失败恢复、人工介入。
单个大模型负责所有事情效率并不高，拆成多个子任务交给不同 Worker 更稳。
开发者真正需要的是“可控的执行系统”，不是一个会聊天但不落地的 AI。
所以你的系统要强调的是：任务拆解能力、调度能力、执行能力、控制能力、审计能力。
3. 总体架构
Layer 1：接入层
终端 CLI：主入口，适合高频开发和调试。
飞书移动端：控制面，适合通知、审批、打断、改路线。
Layer 2：Meta 层
Workflow Designer：把任务变成 Blueprint。
Role Designer：发现角色缺口时生成新的角色提案。
Layer 3：Role Layer
Role Registry：维护 system/user/dynamic roles。
Worker Factory：根据角色实例化执行器。
Layer 4：编排引擎
DAG 解析、Stage 调度、并行分发、失败重规划、checkpoint。
Layer 5：执行层
Claude CLI Worker、Codex CLI Worker，作为本地执行后端。

Layer 6：产物与审计层

4. 任务执行链路
用户输入一个大任务。
Workflow Designer 生成 Blueprint。
Engine 读取 Blueprint，构建执行队列。
Splitter 把复杂 Stage 拆成多个子任务。
Worker Pool 并行或串行执行这些子任务。
Merger 把多个子结果合并成最终产物。
失败时 Re-plan，必要时修改蓝图并重新执行。
最终把项目目录、日志、回放和统计全部保存下来。
5. 大任务拆成子任务
这是你项目里最关键的一层。你不是做一个纯“sub-agent 宿主”，而是做一个受控的子任务分发系统。任务被拆分之
后，子任务会进入 Worker Pool，每个 Worker 拿到自己的上下文、工具白名单和执行目标，彼此隔离，互不污染。
常见拆分方式包括按阶段拆、按领域拆、按风险拆、按并行拆。
6. CLI Worker：Claude CLI / Codex CLI 作为执行后端
你已经明确不想自己直连模型 API，这个决定非常正确。最适合你的做法是：把 claude 和 codex 这些 CLI 当成
Worker backend。它们自己读本地配置文件、环境变量、模型设置、沙盒策略、上下文压缩逻辑，而你的系统只负责
启动它们、喂任务、收结果。
你不用重复实现 provider 兼容、工具协议和上下文策略；你直接借它们成熟的工程能力，比如沙盒、对话、模型切
换、上下文压缩。
7. Role Layer：角色层的作用
Role 层解决的是“谁来干活”的问题。Role 是逻辑身份，Worker 是具体实例。一个 Role 可以实例化成多个 Worker，
因此在并行阶段你可以把同一个“码农角色”复制出多个执行器，分别完成不同子任务。
角色来源分为 System Role、User Role、Dynamic Role 三类。
8. 飞书移动端控制面
飞书不是执行引擎，而是你的控制面。你要让手机成为一个能随时操控系统的入口：看进度、点按钮、发消息、改路
线、确认动态角色、暂停和恢复任务。
飞书适合做任务通知、Blueprint 审批、Dynamic Role 确认、Stage 失败后的重试/跳过/改参数以及任务路线变更。

9. 可视化与手机操控体验
你想要的“可视化更好”并不是做一个重型 Web 编排器，而是做一个轻量移动端可视化控制体验。用户在飞书里看到任务
树、当前 Stage、子任务状态、失败原因、产物链接，必要时可以直接点按钮改变任务路线。
10. Re-plan：失败后的重规划
模型执行不可能总是成功，所以系统必须支持重规划。一个 Stage 失败后，错误日志会回传给 Meta 层或局部 Re-
plan 逻辑，由系统决定是补环境检查、补依赖安装、修改 Stage、还是回到 Designer 重做蓝图。
11. Replay / Artifact / 审计
blueprint.json
history.json
event.log
metrics.json
各 Stage 的文档、代码和测试结果
Replay 的意义不是“好玩”，而是让你能复盘、调试、展示和做简历素材。
12. 推荐的开发阶段
Phase 1：最小内核。Phase 2：Meta-Workflow。Phase 3：子任务拆分与 Worker Pool。Phase 4：CLI
Worker。Phase 5：飞书控制面。Phase 6：可观测和回放。
13. 最终效果
最后做出来的不是一个聊天框，而是一个可以让你输入任务、自动拆解、自动派工、自动跑代码、自动测试、自动修
复、还能在手机上实时接管的任务执行系统。
14. 简历能怎么写
设计并实现终端优先的多模型任务编排系统，支持 AI 自动生成 Workflow Blueprint、子任务拆分、并行执
行与失败重规划。
基于 Claude CLI / Codex CLI 构建本地 Worker 后端，复用其沙盒、上下文压缩和模型切换能力，统一由
外层 Engine 调度。
通过飞书构建移动端控制面，实现任务通知、审批、路线变更和人工干预闭环。
实现产物隔离、事件日志、历史回放与审计能力，使任务执行过程可复现、可追踪、可展示。
15. 最后一句话
这个项目的核心不是“我接了什么模型”，而是“我做了一个能够把复杂任务拆开、派工、执行、修复、回放的 AI 任务执
行平台”。

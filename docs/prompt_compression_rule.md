# Prompt Compression Rule（提示词压缩规则）

> **版本**: v1.0  
> **日期**: 2026-06-15  
> **状态**: 正式设计原则（生效中）

---

## 一、问题背景

本系统采用"外部控制器 + 本地 CLI 执行体"的双层架构：

- **外部控制器**（Python 代码）：负责规划、状态管理、错误分析、循环调度
- **本地 CLI 执行体**（DeepSeek via Claude CLI）：负责写代码、改文件

在实际运行中，我们发现一个高频失败模式：

> 当把长 Traceback、完整的业务描述或混合中英文的重上下文直接发给本地 CLI 执行体时，  
> 模型会退化为"聊天模式"——只在控制台输出解释性文字，而不调用文件写入工具。  
> 结果是：退出码为 0，但目标文件没有任何改动（静默失败）。

---

## 二、核心规则定义

### 规则名称
**Prompt Compression Rule（提示词压缩规则）**

### 一句话定义
> 不要把长上下文直接投喂给本地 CLI 执行体；  
> 由外部控制器先将复杂信息压缩为一句话动作指令，再交给执行体执行。

### 强制约束（Mandatory）

| 编号 | 约束 |
|------|------|
| PCR-1 | 发给本地 CLI 执行体的指令必须是单句话，不超过 3 行 |
| PCR-2 | 指令必须包含触发工具调用的关键词（`must use your write/edit tool`、`do not explain`） |
| PCR-3 | 不得在执行指令中包含完整 Traceback 或超过 5 行的错误输出 |
| PCR-4 | 不得在执行指令中包含请求批准的措辞（`请确认`、`Can you please`、`是否同意`） |
| PCR-5 | 执行指令必须用英文写（DeepSeek CLI 英文指令下工具调用稳定性更高） |

---

## 三、实现方式（两步翻译法）

```
┌──────────────────────────────────────────────────┐
│              外部控制器（Python 代码）             │
│                                                  │
│  Step 1: 翻译（无文件权限）                        │
│  ┌────────────────────────────────────────────┐  │
│  │ claude -p "把这段报错翻译成一句动作指令..."  │  │
│  │  输入：长 Traceback / 复杂业务描述           │  │
│  │  输出：一句话单行指令                        │  │
│  └────────────────────────────────────────────┘  │
│                      │                           │
│                      ▼                           │
│  Step 2: 执行（带 bypassPermissions 权限）         │
│  ┌────────────────────────────────────────────┐  │
│  │ claude -p "[一句话指令]"                    │  │
│  │          --permission-mode bypassPermissions│  │
│  │  输入：极简的单行指令                        │  │
│  │  输出：直接调用工具修改目标文件               │  │
│  └────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────┘
```

### 代码位置

此规则在以下位置实现：

- `polyglot_ai/v1_worker.py` — `main()` 初始填充阶段（`query_claude_cli` + `call_claude_cli`）
- `polyglot_ai/v1_worker.py` — `main()` 自愈修复循环（同上模式）

## 四、为什么这样设计

### 类比
本地 CLI 执行体是一个**专注的执行者**，最佳工作状态是收到一个清晰的命令然后立刻动手。  
如果你给它一份 100 行的报错日志和背景故事，它会变成一个**思考者**，开始解释、分析、询问。

外部控制器（Python 代码）充当**"翻译官 + 大脑皮层"**：看得懂复杂信息，只把最精华的一句话下达给执行体。

### 与 Claude Fable 5 系统的对应关系
Claude Fable 5 通过超长 system prompt 给 Claude 注入大量约束，让 Claude 自己具备"归纳上下文、简化执行"的能力。

我们的做法是：**把这层能力外化到 Python 代码层**，用两次独立的 CLI 调用分离"理解"和"执行"。  
这使得我们对本地 CLI 执行体的要求更低，同时让外部控制器对执行过程有更强的掌控力。

## 五、延伸场景（Future Use）

此规则不只适用于修复 Bug，所有"把复杂信息转化为工具调用"的场景都应遵循：

| 场景 | 压缩前 | 压缩后（执行指令）|
|------|--------|-----------------|
| 初始代码生成 | 完整项目需求 + 文件描述 | "Please implement `date_diff()` in `date_converter.py` using `abs((d1-d2).days)`." |
| Bug 修复 | 完整 pytest 输出 | "Please fix the `date_diff` function in `date_converter.py` by wrapping return value with `abs()`." |
| 重构任务 | 旧代码 + 新架构描述 | "Please refactor `parse_csv()` to use `csv.DictReader` instead of manual split." |

---

## 六、违反此规则的后果

违反 PCR-1 到 PCR-5 任意一条，可能导致：

| 后果 | 表现 |
|------|------|
| **静默失败** | CLI 退出码为 0，但文件未被修改（最常见） |
| **权限请求卡死** | CLI 在等待人工确认，进程挂起 |
| **无效输出** | CLI 输出解释性文字，没有调用文件工具 |
| **上下文混淆** | 模型切换到聊天模式，不再使用工具 |
